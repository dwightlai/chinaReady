# China Ready 外国游客赴华旅行操作指南平台
## 系统设计说明书（V1.0）

**文档状态：** 初稿  
**目标形态：** 移动优先独立网站 + PWA  
**目标用户：** 计划来华或正在中国旅行的国际自由行游客、商务访客  
**首发语言：** 英文  
**首发区域：** 全国通用问题 + 广州深度验证，逐步扩展北京、上海、深圳等城市  
**首发平台：** Web、PWA  
**暂不建设：** 原生 iOS/Android App、微信小程序、浏览器插件、在线交易与预订平台  

---

# 1. 项目背景

外国游客来中国旅行时，真正影响旅行体验的往往不是“有哪些景点”，而是以下高频操作问题：

- 支付宝、微信支付如何注册、实名认证和绑定境外银行卡；
- 中国境内使用哪个地图更准确，是否支持英文；
- 高铁票怎么买，护照信息怎么填写，应该提前多久到站；
- 景区预约是否支持外国护照；
- 酒店是否能够正常办理外国人入住；
- 餐馆没有英文菜单时如何点餐；
- 中国法定节假日对交通、景区和住宿有何影响；
- 地铁出口、景区入口、出租车下车点如何确认；
- 小红书、抖音等平台上的热门地点是否适合外国游客；
- 中国本地推荐的餐厅、商店和网红地点，外国游客是否真正能够找到、预约、支付和使用。

现有英文中国旅游内容主要集中于景点介绍、传统路线和酒店推荐，无法完整解决外国游客在中国数字支付、实名预约、交通出行、中文地址和本地生活服务方面的执行问题。

本项目不建设传统旅游门户，而是建设一套面向国际游客的：

> **China Travel Operating Manual——中国旅行操作说明书。**

平台以“问题解决、操作步骤、失败排查、城市差异、实地核验和持续更新”为核心，帮助用户在出发前完成准备，并在旅途中快速解决实际问题。

# 2. 产品定位

## 2.1 核心定位

**英文定位：**

> Everything you need to make China actually work.

**产品说明：**

> A practical, verified and continuously updated operating guide for international visitors traveling in China.

平台不重点回答“去哪里”，而重点回答：

- 能不能用；
- 怎么设置；
- 需要准备什么；
- 哪一步最容易失败；
- 失败后怎么办；
- 不同城市是否存在差异；
- 信息最近什么时候被验证；
- 应该向中国工作人员展示什么中文内容。

## 2.2 产品差异化

平台与普通旅行博客、旅游门户和AI行程生成器的主要区别如下：

| 对比维度 | 普通旅游网站 | 本平台 |
|---|---|---|
| 核心内容 | 景点、酒店、线路 | 支付、地图、高铁、预约、入住、点餐等操作问题 |
| 内容形式 | 长篇文章 | 结构化操作卡、检查清单、失败排查 |
| 更新方式 | 发布后低频更新 | 明确核验日期、来源和版本 |
| 用户场景 | 出发前阅读 | 出发前准备 + 旅途中现场使用 |
| 城市差异 | 内容较少 | 全国规则与城市差异分离 |
| 失败处理 | 一般不覆盖 | 常见失败、原因和备用方案 |
| 中文支持 | 普通翻译 | 中文地址卡、沟通卡、搜索关键词 |
| 信息可信度 | 编辑经验 | 官方来源、实测、游客反馈分级 |
| 离线能力 | 通常无 | PWA收藏和离线旅行包 |

# 3. 建设目标

## 3.1 第一阶段目标

第一阶段建设一个可被Google等搜索引擎发现、适合手机使用、能够逐步积累用户反馈的英文独立网站。

主要目标：

1. 建立外国游客赴华高频问题数据库；
2. 提供结构化、可执行的英文操作指南；
3. 建立来源、核验、版本和更新机制；
4. 支持移动端访问和PWA离线使用；
5. 通过SEO获得自然流量；
6. 验证用户最关注的问题和付费意愿；
7. 为后续城市旅行包、行程检查和原生App提供数据基础。

## 3.2 非目标

第一阶段不建设：

- 大而全的中国景点百科；
- 在线机票、酒店、高铁票交易系统；
- 大规模用户社区；
- 实时聊天客服中心；
- 原生iOS和Android应用；
- 实时地图导航；
- 在线翻译引擎；
- 自动抓取并翻译小红书、携程内容；
- 基于用户位置的实时推荐；
- 全自动AI旅行规划平台。

# 4. 用户分析

## 4.1 目标用户

### A. 第一次来中国的国际自由行游客

主要需求：

- 出发前安装和配置必要App；
- 确认支付和上网方式；
- 学会购买高铁票；
- 确认酒店和景区是否支持护照；
- 解决地图、交通和语言问题。

### B. 已在中国旅行的游客

主要需求：

- 支付失败处理；
- 找不到正确入口和地址；
- 点餐、打车、进站、入住沟通；
- 离线查看攻略；
- 快速展示中文沟通卡。

### C. 来华商务访客

主要需求：

- 机场到酒店和展馆交通；
- 支付、通信、发票、用车；
- 商务地点中文地址；
- 广交会、展会、工厂拜访配套信息；
- 商务行程之外的短途旅行建议。

### D. 海外旅行内容创作者和旅行顾问

主要需求：

- 获取可核验的赴华操作信息；
- 获取城市差异和最新变化；
- 引用权威来源；
- 为客户或粉丝制作赴华准备材料。

# 5. 核心业务原则

## 5.1 可执行优先

所有指南必须能够回答：

- 用户现在需要做什么；
- 需要准备哪些材料；
- 在哪个App或页面完成；
- 成功后如何验证；
- 失败后如何处理；
- 是否有备用方案。

## 5.2 来源可追溯

关键规则和政策必须关联来源，并标识来源类型：

- 官方来源；
- 平台官方说明；
- 编辑实测；
- 合作伙伴确认；
- 多名游客反馈；
- 单一游客反馈；
- 尚未核实线索。

## 5.3 更新时间透明

每个操作指南必须展示：

- 首次发布时间；
- 最近修订时间；
- 最近核验时间；
- 核验方式；
- 当前状态；
- 是否可能因城市、银行卡、国籍或App版本而变化。

## 5.4 事实与经验分离

以下信息必须分开呈现：

- 官方规则；
- 编辑实测；
- 游客经验；
- 平台推测；
- AI生成的整理结果。

AI不能被作为唯一事实来源。

## 5.5 不直接搬运第三方内容

小红书、携程、大众点评、Reddit等平台主要用于：

- 发现问题；
- 发现新地点；
- 收集失败案例；
- 获取用户反馈线索；
- 判断近期变化。

平台不得：

- 批量抓取并翻译原文；
- 复制第三方图片、视频和路线图；
- 逐篇翻译并重新发布；
- 长期展示未经授权的价格、评分和库存；
- 将AI改写视为版权免责。

# 8. 核心功能设计

# 8.1 高频问题指南

## 8.1.1 指南基本结构

每个指南包含：

1. 问题标题；
2. 一句话当前结论；
3. 适用人群；
4. 使用前提；
5. 分步操作；
6. 操作截图；
7. 成功验证方法；
8. 常见失败；
9. 失败原因；
10. 备用方案；
11. 城市差异；
12. 相关指南；
13. 官方来源；
14. 实测记录；
15. 最近核验日期；
16. 用户反馈入口。

## 8.1.2 指南状态

- Draft：草稿；
- Under Review：审核中；
- Verified：已核验；
- Partially Verified：部分核验；
- Outdated：可能过期；
- Suspended：暂不推荐；
- Archived：已归档。

## 8.1.3 核验等级

| 等级 | 含义 |
|---|---|
| A | 官方来源 + 编辑或合作伙伴实测 |
| B | 官方来源已确认，尚未实测 |
| C | 多名近期游客反馈一致 |
| D | 单一用户反馈或待核实线索 |

关键支付、交通、签证、铁路和预约内容原则上不得仅以D级信息公开为确定结论。

# 8.2 出发前检查清单

用户可以建立个人检查清单。

主要事项：

- 支付宝已安装；
- 支付宝已完成实名认证；
- 境外银行卡已绑定；
- 微信支付已配置；
- 已准备备用支付方式；
- eSIM或漫游已准备；
- 已安装可用地图；
- 已安装翻译工具；
- 高铁账号已注册；
- 酒店已确认可使用护照登记；
- 已避开重大节假日；
- 紧急联系人已保存。

系统应支持：

- 勾选完成；
- 保存进度；
- 发送邮件提醒；
- 根据未完成事项推荐指南；
- 导出PDF或PWA离线清单。

# 8.4 城市差异

同一问题可以包含全国通用规则和城市补充。

示例：

**问题：外国游客如何乘坐地铁**

全国通用内容：

- 可用支付方式；
- 安检要求；
- 行李注意事项；
- 地铁运营基本规则。

广州补充：

- 广州地铁支付方式；
- 常见换乘站；
- 广交会期间拥挤时段；
- 机场南站和机场北站区别。

北京补充：

- 景区集中区域换乘；
- 安检耗时；
- 热门站点拥挤风险。

城市差异不得复制整篇指南，而应通过关联方式继承全国内容并增加覆盖项。

# 8.8 PWA与离线能力

第一阶段PWA功能：

- 添加到主屏幕；
- 缓存首页和基础导航；
- 收藏指南离线访问；
- 中文沟通卡离线使用；
- 检查清单离线勾选；
- 已购买旅行包离线下载；
- 网络恢复后同步数据；
- 显示离线内容版本和更新时间。

离线缓存原则：

- 仅缓存用户主动收藏或下载的内容；
- 政策类内容必须显示缓存时间；
- 网络恢复后检查是否存在更新；
- 过期内容必须提示重新同步。

```text
International Visitor
        │
        ▼
Mobile-first Website / PWA
        │
        ├── Public Content Pages
        ├── Search
        ├── Readiness Test
        ├── Checklist
        ├── Offline Packs

        │
        ▼
Application API
        │
        ├── Content Service
        ├── Guide Service
        ├── Place Service
        ├── Verification Service
        ├── Feedback Service
        ├── User Service
        ├── Commerce Service
        └── Search Service
        │
        ├── PostgreSQL
        ├── Object Storage
        ├── Search Index
        └── Cache
        │
        ▼
Admin Management System
        ├── Content Editing
        ├── Source Review
        ├── Verification Workflow
        ├── Feedback Review
        ├── SEO Management
        └── Product Management
```

## 12.2 架构原则

- 第一阶段单体优先；
- 不使用Spring Cloud微服务；
- 前后台分离；
- 内容页支持服务端渲染或静态生成；
- SEO优先；
- 移动优先；
- PWA逐步增强；
- 数据模型支持多语言，但首发只维护英文；
- 所有关键内容保留来源和版本；
- AI只作为编辑辅助，不直接无人审核发布。

# 18. 合规与风险控制

## 18.1 版权

- 不直接翻译并发布第三方完整文章；
- 不复制小红书、携程等平台图片；
- 图片优先使用自拍、授权图库或合作伙伴素材；
- 引用必要片段时控制范围并标明来源；
- 关键内容尽量回到官方来源；
- 对用户上传内容取得必要授权。

## 18.2 内容责任

- 明确内容为旅行信息辅助；
- 政策可能变化；
- 用户应以官方最新规则为准；
- 重要内容显示最后核验时间；
- 对无法确认的信息明确标识不确定性；
- 不承诺所有银行卡、手机号或账户均能成功。

## 18.3 隐私

最小化收集：

- 邮箱；
-收藏；
-检查清单；
-购买记录；
-用户反馈。

原则上不收集：

- 护照照片；
- 银行卡完整信息；
- 支付密码；
- 精确长期位置轨迹；
- 签证文件。

## 18.4 用户反馈风险

- 用户反馈先审核后公开；
- 隐去姓名、订单号和个人信息；
- 不公开未验证的商家指控；
- 不使用单个反馈形成确定性结论；
- 建立纠错和投诉机制。

- 主要内容页首屏加载目标小于2秒；
- 移动端优先；
- 图片自动压缩；

- 后台二次认证；
- 权限分级；
- 操作日志；
- 上传文件类型限制；
- API限流；
- 防止XSS、CSRF和SQL注入；
- 定期依赖扫描；
- 数据库备份和恢复演练。

## 19.4 可维护性

- 模块化单体；

# 25. 主要风险与应对

| 风险 | 应对 |
|---|---|
| 信息更新快 | 核验日期、复核任务、用户纠错 |
| 内容被AI搜索替代 | 结构化操作卡、实测、失败排查、城市差异 |
| SEO增长慢 | 先做长尾高频问题和工具型页面 |
| 小红书等来源版权风险 | 仅做线索，官方核验后原创撰写 |
| 用户反馈不准确 | 审核、聚类、多来源确认 |
| 全国范围过大 | 先全国通用 + 广州深度 |
| 产品做成普通博客 | 强化检查清单、评分、离线卡和地点可用性 |
| 原生App投入过早 | 网站和PWA先验证使用频率 |
| 用户不愿付费 | 先销售Arrival Pack和人工行程检查 |
| 企业软件思维过重 | 前台保持轻量，后台按内容治理设计 |

# 26. 最终建设建议

本项目不应被定义为“中国旅游英文攻略站”，而应定义为：

> **外国游客赴华旅行操作指南与可用性验证平台。**

第一阶段的产品核心不是内容数量，而是建立以下能力：

1. 高频问题结构化；
2. 操作步骤可执行；
3. 失败场景可排查；
4. 全国规则与城市差异分离；
5. 信息来源可追溯；
6. 内容最近核验时间透明；
7. 中文地址和沟通卡可离线使用；
8. 用户反馈能够推动持续更新。

建议首个版本围绕：

> **“外国游客第一次来中国，如何完成支付、上网、地图、高铁、酒店和景区预约”**

进行建设。

技术形态采用：

> **Next.js移动优先网站 + PWA + Spring Boot模块化单体 + PostgreSQL + 内容核验后台。**

不应在首版建设原生App、复杂旅行规划、实时地图、社区和大规模预订平台。

项目的长期竞争力不来自翻译，而来自：

> **本地信息发现、官方核验、实地验证、外国游客适配、失败排查和持续更新。**
