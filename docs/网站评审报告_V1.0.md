# China Trip Check 网站评审报告

**评审日期：** 2026-07-14  
**评审范围：** 线上网站 chinatripcheck.com + 工作空间源码 + 设计文档  
**评审维度：** 内容完整性、布局与交互、技术实现、与设计文档的偏差

---

## 一、总体评价

| 维度 | 评分 | 说明 |
|---|---|---|
| 产品定位 | ★★★★★ | "风险检查器"定位清晰，与攻略站形成明确差异化 |
| 内容覆盖 | ★★★☆☆ | 四个核心工具+8篇指南已上线，但规则深度不足，关键功能缺失 |
| 布局设计 | ★★★★☆ | 移动优先、视觉层次清晰、配色系统完整 |
| 交互体验 | ★★★★☆ | 问卷流流畅，条件显示、进度条、本地存储都到位 |
| 技术质量 | ★★★★☆ | 架构干净、类型安全，但规则引擎有逻辑缺陷 |

## 二、内容评审

### 2.1 做得好的部分

| 项目 | 评价 |
|---|---|
| 产品定位 | "Find the risks before they disrupt your China trip" —— 一句话讲清楚核心价值，不堆砌景点信息 |
| 四个检查工具 | 完全对齐设计文档优先级：Readiness(总入口) → Payment → Dates → Hotel Arrival |
| 文案风格 | 英文简洁有力，按钮文案行动导向（"Check payments"、"See my report"） |
| 信任信号 | Trust Strip 三点设计到位：Private by design / Clear reasoning / Sources reviewed |
| 指南体系 | 8篇指南覆盖支付、日期、酒店三大方向，与工具关联 |
| 风险报告结构 | Score → Metrics → Findings → Actions → Backup Plan → Disclaimer，层次清晰 |
| 保守措辞 | 结果页免责声明、how-it-works 页面"what this tool cannot do"段落，符合设计文档的保守判断原则 |

### 2.2 内容缺陷

| # | 问题 | 严重度 | 说明 |
|---|---|---|---|
| C1 | **节假日风险窗口未生效** | 严重 | `preRiskDays`/`postRiskDays` 字段定义了但规则完全没使用。date-overlaps 只比对官方假期日期，出发前/节后风险窗口未扩展。设计文档明确要求节前风险检测。 |
| C2 | **多节日重叠只显示一个** | 严重 | 所有节日规则共享 group `holiday-overlap`，merge-findings 会合并为一。跨数月行程只能看到第一个匹配的节日名称，后续节日被吞掉。 |
| C3 | **缺少广交会事件** | 中 | 设计文档将广交会列为首发城市事件，holiday-events.ts 中完全没有。 |
| C4 | **缺少春运/暑期** | 中 | 设计文档将春运和暑期列为独立风险事件，当前只有春节法定假期。 |
| C5 | **规则数量未达标** | 中 | 设计文档要求 Readiness 25-35条规则，实际只有9条；Payment 要求15-20条，实际9条。规则深度不足导致检查结果偏粗糙。 |
| C6 | **缺少邮件保存功能** | 中 | 设计文档MVP明确包含"邮件保存结果"，源码中无任何邮件捕获逻辑。 |
| C7 | **缺少联盟链接** | 低 | 设计文档第一阶段商业化包含eSIM/酒店/Trip.com联盟，当前无任何联盟链接。 |
| C8 | **指南数量偏少** | 低 | 当前8篇，设计文档目标12-16篇。缺少高铁方向4篇指南（合理，因第二阶段才做）。 |
| C9 | **无统计数据** | 低 | 设计文档提到Plausible/GA，源码中未见统计代码。 |
| C10 | **DATES_OUTSIDE_WINDOW 逻辑不完整** | 中 | 只检查 arrivalDate 是否在2026年范围内，未检查 departureDate。12月底到达、1月离开的行程不会被标记为超出验证窗口。 |

### 2.3 首页内容评价

首页首屏文案与设计文档 §15.1 基本一致，但有两处偏差：

| 设计文档要求 | 实际实现 | 差异 |
|---|---|---|
| "Check payments, dates, hotel arrival and train risks" | "Check your payments, travel dates and hotel arrival" | 去掉了 train risks（合理，因第二阶段才做） |
| 四个入口按钮文案 "Check My China Readiness" 等 | "Start full check" / "Check payments" 等 | 更简洁，可接受 |

首页 hero 区域的 Readiness Score 预览卡片（64分、2 Critical、3 High、4 Ready）是硬编码的 demo 数据——作为视觉钩子可以，但需要确保不会被误认为是真实结果。

## 四、技术实现评审

### 4.1 架构亮点

```
configs (TypeScript配置)
  → types (严格类型定义)
  → operators (条件求值引擎)
  → evaluate (规则匹配)
  → merge-findings (同组合并)
  → score (评分计算)
  → risk-report (结果渲染)
```

- **规则引擎设计合理**：AND/OR 条件组合、12种操作符（eq/neq/includes/missing/gt/lt/date-before/date-after/date-overlaps/time-between等）
- **Finding 合并逻辑**：同 group 的规则合并为一条，取最高严重度，actions 去重——避免重复提示
- **类型安全**：Severity、CheckSlug、Operator 等均为字面量联合类型，编译期可捕获错误
- **SSG 支持**：`generateStaticParams` 预生成四个检查页
- **SEO 基础**：sitemap.ts、robots.ts、JSON-LD、OpenGraph metadata
- **localStorage 防御性编程**：版本号校验、类型守卫、异常时自动清理

### 4.2 技术问题

| # | 问题 | 严重度 | 位置 | 说明 |
|---|---|---|---|---|
| T1 | **preRiskDays/postRiskDays 未被使用** | 严重 | dates.ts L5-22 | HolidayEvent 定义了这两个字段，但 holidayRules 只用 startDate/endDate 构造 date-overlaps 条件。节前/节后风险窗口完全失效。 |
| T2 | **多节日合并导致信息丢失** | 高 | dates.ts L9 | 所有节日规则 group="holiday-overlap"，merge-findings 合并后只显示第一个匹配节日的 title。应改为每个节日独立 group 或在 explanation 中列出所有匹配节日。 |
| T3 | **评分模型过于简单** | 中 | score.ts L3-8 | 固定扣分（critical=24, high=12），不考虑规则权重、概率、可恢复性。设计文档§13.2提到 severityWeight/probabilityWeight/recoverabilityWeight/urgencyWeight，但未实现。 |
| T4 | **information 严重度在 ReportMetrics 中不显示** | 中 | report-metrics.tsx L5-8 | metrics 数组只有 Critical/High risk/Ready，缺少 Information 计数。但 Finding 组件会渲染 information 级别的风险项。 |
| T5 | **无 ErrorBoundary** | 低 | check-experience.tsx | 客户端组件无错误边界，localStorage 异常或规则求值异常可能导致白屏。 |
| T6 | **日期边界处理** | 低 | operators.ts L14 | `dateValue` 使用 `Date.parse` + UTC，可能在某些时区下日期边界判断有偏差。 |

### 4.3 测试覆盖

工作空间存在 `tests/` 目录（17个文件）和 `playwright.config.ts`、`vitest.config.ts`。测试框架已搭建，需确认测试覆盖率是否覆盖规则引擎的核心逻辑。

## 七、结论

China Trip Check 的**产品骨架和用户体验做得很好**——定位清晰、视觉干净、问卷流畅、代码架构规范。它成功地把"中国旅行风险检查器"这个概念落地为一个可用产品。

但**离设计文档承诺的MVP还有距离**：规则深度不足（9条 vs 25-35条）、节假日风险窗口逻辑有缺陷、结果页缺少核心的"下一检查"引导路径、邮件保存/联盟链接等MVP功能未实现。

**建议的修复优先级：** 先修P0（3项，影响核心正确性），再补P1（5项，补齐MVP），最后P2按需优化。P0的3项修复工作量都不大，但直接决定了产品是否"可用且正确"。
